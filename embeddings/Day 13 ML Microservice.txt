Day 13 — ML Microservice Deep Dive

First — Why Do We Even Need an ML Service?
This is the most important question to answer before writing a single line.
Your recommendation system already has a user profile in Redis that looks like this:
json{
  "genreAffinities": {
    "crime": 0.74,
    "thriller": 0.61,
    "comedy": 0.12
  }
}
So the naive question is — why not just query PostgreSQL directly?
sqlSELECT * FROM content 
WHERE genre IN ('crime', 'thriller') 
ORDER BY release_year DESC 
LIMIT 20
That works. But it has fundamental problems:
Problem 1 — It only uses genre. A user who loves crime dramas might hate crime comedies. Genre alone doesn't capture taste. A real recommendation needs to understand combinations of attributes — genre + tone + cast + pacing.
Problem 2 — It ignores relationships between users. If user_001 and user_005 have very similar watch histories, and user_005 recently loved a new sci-fi thriller, user_001 would probably love it too. A SQL query has no concept of user similarity.
Problem 3 — It doesn't learn. A hardcoded SQL query returns the same results regardless of what 10,000 users collectively watched last week. An ML model improves as more data flows through it.
Problem 4 — It can't rank by predicted enjoyment. SQL can sort by release year or popularity. An ML model sorts by predicted probability that this specific user will enjoy this specific item. That's a fundamentally different and more powerful operation.
So the ML service exists to answer one question that SQL cannot:

"Given everything we know about this user's taste, rank these 100 content items by how much this specific user is predicted to enjoy each one."

That's the job of the ML service. Nothing more, nothing less.

What the ML Service Actually Does in This System
Before talking about datasets and models, understand the exact role:
Input:  User profile (genre affinities, watch history)
        Content catalog (100 items with metadata)

Output: Ranked list of contentIds
        [most predicted enjoyment → least predicted enjoyment]
It is a scoring and ranking service. It takes candidates (your content catalog) and ranks them for a specific user. It is not a search engine. It is not a discovery engine. It is a ranker.
This distinction matters because your Spring Boot service handles everything else:

Fetching the candidates (content catalog) → Spring Boot
Caching the result → Spring Boot
Fallback when ML is slow → Spring Boot
A/B variant routing → Spring Boot

The ML service is a narrow, focused component. Keep it that way.

Why a Separate Python Service and Not Java?
Three practical reasons:
The ML ecosystem lives in Python. Libraries like scikit-learn, pandas, surprise, implicit — these have no Java equivalents of comparable maturity. Trying to do ML in Java is swimming upstream.
Deployment independence. Your ML model might need to be retrained weekly as new data comes in. If it's inside your Spring Boot service, retraining requires redeploying the entire application. As a separate service, you retrain and redeploy only the ML container.
Language-appropriate scaling. ML inference is CPU-heavy. REST handling is I/O-bound. These scale differently. Separating them lets you scale each independently.

What Models to Consider
Now the interesting part. There are three families of recommendation algorithms. Understand all three conceptually. Implement one.

Family 1 — Content-Based Filtering
Core idea: Recommend items similar to what the user already liked, based on item attributes.
How it works:
Every content item is represented as a feature vector:
c_001 = [crime:1, thriller:1, action:0, comedy:0, drama:0, ...]
c_002 = [crime:0, thriller:0, action:0, comedy:1, romance:1, ...]
Every user has a profile vector (their genre affinities):
user_001 = [crime:0.74, thriller:0.61, action:0.20, comedy:0.12, ...]
The score for user_001 on content c_001 is the dot product of these two vectors:
score = (0.74 × 1) + (0.61 × 1) + (0.20 × 0) + (0.12 × 0)
      = 0.74 + 0.61
      = 1.35
Higher score = better match. Rank all content by score, return top 20.
Pros: Simple to implement, explainable ("you liked crime content, here's more crime content"), works for new users if you have even minimal profile data.
Cons: Creates a "filter bubble" — users only ever see more of what they already like. Doesn't discover new tastes.
This is what your Day 13 plan already implements. It's the right choice for the prototype.

Family 2 — Collaborative Filtering
Core idea: Recommend items that similar users enjoyed, regardless of item attributes.
How it works:
Build a user-item matrix:
         c_001  c_002  c_003  c_004
user_001   4.5    0      3.0    0
user_002   4.0    0      0      5.0
user_003   0      4.5    0      3.5
user_004   3.5    0      4.0    0
Find users with similar rating patterns (user_001 and user_002 both rated c_001 highly → they're similar). Recommend to user_001 the items that user_002 rated highly but user_001 hasn't seen yet → c_004.
Two approaches:
Memory-based (k-Nearest Neighbors): Find the k most similar users, aggregate their ratings. Simple but doesn't scale.
Model-based (Matrix Factorization): Decompose the user-item matrix into two smaller matrices (user factors × item factors). The product of a user's factors and an item's factors predicts the rating. This is what Netflix Prize solutions used.
Library to use: scikit-surprise — specifically designed for collaborative filtering.
pythonfrom surprise import SVD, Dataset, Reader
from surprise.model_selection import train_test_split

reader = Reader(rating_scale=(0, 1))
data = Dataset.load_from_df(ratings_df[['userId','contentId','rating']], reader)
trainset, testset = train_test_split(data, test_size=0.2)

model = SVD(n_factors=50, n_epochs=20)
model.fit(trainset)

# Predict score for user_001 on content c_045
prediction = model.predict('user_001', 'c_045')
print(prediction.est)  # predicted rating
Pros: Discovers cross-genre recommendations. Captures taste patterns humans can't articulate. More accurate than content-based for users with rich history.
Cons: Cold start problem — new users with no history get no recommendations. Requires enough data to find similar users. Harder to explain ("why am I seeing this?").

Family 3 — Hybrid (Content + Collaborative)
Core idea: Use both. Score each item with both methods, combine the scores.
pythonfinal_score = (0.6 × collaborative_score) + (0.4 × content_based_score)
Why 60/40? Collaborative filtering is more accurate for users with history. Content-based provides a safety net for users with sparse history. The weights are tunable — this is literally what your A/B test variants are for.
This maps directly to your A/B test setup:

Variant "content-based" → pure content-based filtering
Variant "popularity" → most-watched globally (baseline)
Variant "hybrid" → 60% collaborative + 40% content-based

Step 3 — The Model (Core — 45 minutes)
python# model.py
import numpy as np
from sklearn.preprocessing import MultiLabelBinarizer
import joblib, os

GENRES = ["crime","comedy","action","romance","thriller",
          "documentary","sci-fi","horror","drama","animation"]

class RecommendationModel:
    
    def __init__(self):
        self.content_catalog = []
        self.content_vectors = {}  # contentId → genre vector
        self.mlb = MultiLabelBinarizer(classes=GENRES)
        self.mlb.fit([GENRES])  # fit on all genres
    
    def load_catalog(self, catalog):
        """Build content feature vectors from catalog"""
        self.content_catalog = catalog
        for item in catalog:
            genres = item.get("genres", [])
            # One-hot encode genres
            vector = self.mlb.transform([genres])[0]
            self.content_vectors[item["contentId"]] = vector.astype(float)
        print(f"Loaded {len(self.content_catalog)} content items")
    
    def content_based_score(self, user_affinities, content_id):
        """Dot product of user affinity vector and content genre vector"""
        if content_id not in self.content_vectors:
            return 0.0
        
        # Build user vector in same order as GENRES
        user_vector = np.array([
            user_affinities.get(genre, 0.1) for genre in GENRES
        ])
        
        content_vector = self.content_vectors[content_id]
        
        # Dot product = similarity score
        return float(np.dot(user_vector, content_vector))
    
    def popularity_score(self, content_id):
        """Simple recency-based score — newer = more popular"""
        for item in self.content_catalog:
            if item["contentId"] == content_id:
                year = item.get("releaseYear", 2020)
                return (year - 2018) / 7.0  # normalize 2018-2025 to 0-1
        return 0.0
    
    def rank_content(self, user_affinities, variant="hybrid", limit=20,
                     recently_watched=None):
        """Main ranking method"""
        recently_watched = recently_watched or []
        scores = []
        
        for item in self.content_catalog:
            content_id = item["contentId"]
            
            # Skip recently watched (don't recommend what they just saw)
            if content_id in recently_watched[-10:]:
                continue
            
            cb_score = self.content_based_score(user_affinities, content_id)
            pop_score = self.popularity_score(content_id)
            
            if variant == "content-based":
                final_score = cb_score
            elif variant == "popularity":
                final_score = pop_score
            else:  # hybrid (default)
                final_score = (0.7 * cb_score) + (0.3 * pop_score)
            
            scores.append({
                "contentId": content_id,
                "score": round(final_score, 4)
            })
        
        # Sort by score descending
        scores.sort(key=lambda x: x["score"], reverse=True)
        
        return [item["contentId"] for item in scores[:limit]]

First — The Core Intuition
Before any math, understand what problem this solves.
You have users and content. Some users have watched some content and you know how much they liked it (completion ratio). Most combinations are unknown — user_001 hasn't watched 95% of your catalog. The matrix looks like this:
           c_001  c_002  c_003  c_004  c_005  c_006
user_001    0.9    0.0    0.7    0.0    0.0    0.8
user_002    0.0    0.8    0.0    0.9    0.0    0.0
user_003    0.0    0.0    0.8    0.0    0.6    0.0
user_004    0.8    0.0    0.0    0.0    0.0    0.9
user_005    0.0    0.7    0.0    0.8    0.0    0.0
The 0.0 entries don't mean the user disliked the content. They mean the user hasn't watched it yet. Your job is to predict what those blank entries would be if the user did watch it — and recommend the ones with the highest predicted scores.
Matrix Factorization answers exactly this question.

The Mathematical Idea (Without Heavy Math)
The insight behind matrix factorization is this:

A user's taste and a content item's characteristics can both be described by a small set of hidden (latent) factors. If a user's factors align with a content item's factors, the user will enjoy it.

What are these latent factors? You don't define them. The algorithm discovers them from the data. They might loosely correspond to things like "gritty vs. light-hearted", "fast-paced vs. slow-burn", "character-driven vs. plot-driven" — but you never label them. They're just numbers.
The decomposition:
You take your large sparse matrix R (users × items) and approximate it as the product of two smaller dense matrices:
R  ≈  U  ×  V^T

R : 1000 users  × 500 items    (sparse, mostly unknown)
U : 1000 users  × 50 factors   (dense, fully populated)
V : 500 items   × 50 factors   (dense, fully populated)
The number 50 is the number of latent factors — a hyperparameter you choose. Common values are 20, 50, 100.
What each matrix means:
U — the User Matrix. Each row is a user represented as 50 numbers. These numbers capture that user's taste in the latent factor space. user_001's row might be [0.8, -0.3, 0.5, 0.1, ...].
V — the Item Matrix. Each row is a content item represented as 50 numbers. These capture the item's characteristics in the same latent factor space.
The prediction:
To predict how much user_001 will enjoy content c_045:
predicted_score = U[user_001] · V[c_045]
                = dot product of their 50-dimensional vectors
                = 0.8×0.3 + (-0.3)×(-0.2) + 0.5×0.7 + ...
                = some number between 0 and 1
High dot product = high alignment = user will probably enjoy this content.
How are U and V learned?
Through an optimization process (gradient descent or SVD) that minimizes the difference between:

The predicted scores (U × V^T)
The actual known scores (the non-zero entries in R)

The algorithm adjusts U and V iteratively until predictions on known entries are accurate. Then you trust it to predict unknown entries.

Why SVD Specifically
SVD stands for Singular Value Decomposition. It's a mathematical technique that finds the optimal decomposition of a matrix into lower-dimensional representations — optimal meaning it captures the maximum variance in the fewest factors.
In the context of recommendations, the library scikit-surprise implements a variant called Funk SVD (named after Simon Funk who published it during the Netflix Prize competition). It's not pure mathematical SVD — it's a gradient descent optimization that finds U and V by minimizing prediction error on known ratings.
This is the algorithm that was at the core of many top Netflix Prize solutions. It's battle-tested, well-understood, and appropriate for a portfolio project.

How MovieLens Fits Into This
MovieLens is essentially a pre-built version of your user-item matrix — already collected, cleaned, and formatted. Instead of waiting for your simulator to generate thousands of events, you bootstrap with real human behavior data.
The mapping from MovieLens to your system:
MovieLens concept          Your system concept
─────────────────────────────────────────────
userId                  →  userId
movieId                 →  contentId
rating (0.5–5.0)        →  completionRatio (0.0–1.0)
genres (pipe-separated) →  genres list
The rating-to-completionRatio conversion:
pythoncompletion_ratio = (rating - 0.5) / 4.5   # maps 0.5→0.0, 5.0→1.0
What MovieLens gives you that your simulator doesn't:
Real human nuance. Real users gave different ratings to content in the same genre. User A gave crime movies 4.5 and 4.0 and 2.0 — because some crime movies resonated and some didn't. A simulator assigns scores based on a hidden preference probability, which is too clean. Real data has noise, contradictions, and surprise — which is exactly what makes collaborative filtering valuable.

MOVIELENS_TO_YOUR_GENRES = {
    "Crime":       "crime",
    "Thriller":    "thriller",
    "Comedy":      "comedy",
    "Romance":     "romance",
    "Action":      "action",
    "Documentary": "documentary",
    "Sci-Fi":      "sci-fi",
    "Horror":      "horror",
    "Drama":       "drama",
    "Animation":   "animation",
    # Many-to-one mappings
    "Adventure":   "action",
    "Fantasy":     "sci-fi",
    "Mystery":     "thriller",
    "War":         "drama",
    "History":     "documentary",
    "Western":     "drama",
    "Music":       "comedy",
    "Musical":     "comedy",
    "Film-Noir":   "crime",
    "Children":    "animation",
    "(no genres listed)": None
}

def map_genres(movielens_genres_string):
    """
    Input:  "Crime|Thriller|Drama"
    Output: ["crime", "thriller", "drama"]
    """
    raw_genres = movielens_genres_string.split("|")
    mapped = []
    for g in raw_genres:
        mapped_genre = MOVIELENS_TO_YOUR_GENRES.get(g)
        if mapped_genre and mapped_genre not in mapped:
            mapped.append(mapped_genre)
    return mapped if mapped else ["drama"]  # fallback

import pandas as pd
import numpy as np

def load_movielens(data_path="ml-latest-small/"):
    
    # Load raw data
    ratings = pd.read_csv(f"{data_path}ratings.csv")
    movies = pd.read_csv(f"{data_path}movies.csv")
    
    print(f"Raw ratings: {len(ratings)}")
    print(f"Raw movies: {len(movies)}")
    
    # Convert ratings to completion ratios (0.5-5.0 → 0.0-1.0)
    ratings["completionRatio"] = (ratings["rating"] - 0.5) / 4.5
    ratings["completionRatio"] = ratings["completionRatio"].round(3)
    
    # Convert IDs to your format
    ratings["userId"] = "ml_user_" + ratings["userId"].astype(str)
    ratings["contentId"] = "ml_" + ratings["movieId"].astype(str)
    
    # Map MovieLens genres to your genres
    movies["mappedGenres"] = movies["genres"].apply(map_genres)
    movies["contentId"] = "ml_" + movies["movieId"].astype(str)
    
    # Filter to only movies that have at least 10 ratings
    # (sparse items make the model worse)
    item_counts = ratings["contentId"].value_counts()
    popular_items = item_counts[item_counts >= 10].index
    ratings = ratings[ratings["contentId"].isin(popular_items)]
    
    # Filter to only users with at least 20 ratings
    # (sparse users also hurt model quality)
    user_counts = ratings["userId"].value_counts()
    active_users = user_counts[user_counts >= 20].index
    ratings = ratings[ratings["userId"].isin(active_users)]
    
    print(f"Filtered ratings: {len(ratings)}")
    print(f"Active users: {len(active_users)}")
    print(f"Active items: {len(popular_items)}")
    
    return ratings, movies

def prepare_for_surprise(ratings_df):

import pandas as pd

from surprise.model_selection import train_test_split, cross_validate

import os

def train_and_save(data_path="ml-latest-small/", model_path="models/svd_model.pkl"):
    
    os.makedirs("models", exist_ok=True)
    
    # Load and prepare data
    ratings, movies = load_movielens(data_path)
    prepared = prepare_for_surprise(ratings)
    
    print("Training SVD model...")
    
    # Configure surprise reader
    # rating_scale matches your completionRatio range (0.0 to 1.0)
    reader = Reader(rating_scale=(0.0, 1.0))
    
    # Load into surprise Dataset
    data = Dataset.load_from_df(prepared, reader)
    
    # Split into train and test
    trainset, testset = train_test_split(data, test_size=0.2, random_state=42)

    # n_epochs:  number of gradient descent iterations
    # lr_all:    learning rate

    model = SVD(
        n_factors=50,
        n_epochs=30,
        lr_all=0.005,
        reg_all=0.02,
        random_state=42,
        verbose=True
    )
    
    # Train
    model.fit(trainset)
    
    # Evaluate on test set
    predictions = model.test(testset)
    rmse = accuracy.rmse(predictions)
    mae = accuracy.mae(predictions)
    print(f"RMSE: {rmse:.4f}")  # root mean squared error
    print(f"MAE:  {mae:.4f}")   # mean absolute error
    
    # Save the trained model
    joblib.dump(model, model_path)
    print(f"Model saved to {model_path}")
    
    # Save the list of known userIds and itemIds
    # (needed to detect unknown users at inference time)
    known_users = set(prepared["userId"].unique())
    known_items = set(prepared["contentId"].unique())
    joblib.dump({"users": known_users, "items": known_items}, 
                "models/known_entities.pkl")
    
    return model, movies

Step 5 — The Hybrid Recommendation Engine
This is the core of your ML service. It combines SVD with content-based filtering.
python# recommender.py
import numpy as np
import joblib
import pandas as pd
from genre_mapping import map_genres

GENRES = ["crime","comedy","action","romance","thriller",
          "documentary","sci-fi","horror","drama","animation"]

class HybridRecommender:
    
    def __init__(self):
        self.svd_model = None
        self.known_users = set()
        self.known_items = set()
        self.content_catalog = []         # list of dicts from Spring Boot
        self.content_genre_vectors = {}   # contentId → numpy array
    
    def load_model(self, model_path="models/svd_model.pkl",
                   entities_path="models/known_entities.pkl"):
        """Load the trained SVD model"""
        self.svd_model = joblib.load(model_path)
        entities = joblib.load(entities_path)
        self.known_users = entities["users"]
        self.known_items = entities["items"]
        print(f"SVD model loaded | known users: {len(self.known_users)} "
              f"| known items: {len(self.known_items)}")
    
    def load_catalog(self, catalog):
        """
        Load content catalog from Spring Boot service.
        Build genre vectors for content-based scoring.
        """
        self.content_catalog = catalog
        
        for item in catalog:
            content_id = item["contentId"]
            genres = item.get("genres", [])
            
            # One-hot encode genres as numpy array
            vector = np.zeros(len(GENRES))
            for genre in genres:
                if genre in GENRES:
                    vector[GENRES.index(genre)] = 1.0
            
            self.content_genre_vectors[content_id] = vector
        
        print(f"Catalog loaded: {len(self.content_catalog)} items")
    
    # ─────────────────────────────────────────────
    # Scoring Methods
    # ─────────────────────────────────────────────
    
    def svd_score(self, user_id, content_id):
        """
        Use SVD model to predict how much user will enjoy content.
        Returns 0.5 (neutral) if user or item is unknown.
        """
        if (self.svd_model is None 
                or user_id not in self.known_users 
                or content_id not in self.known_items):
            return 0.5  # neutral fallback for unknown entities
        
        prediction = self.svd_model.predict(user_id, content_id)
        return float(prediction.est)
    
    def content_based_score(self, genre_affinities, content_id):
        """
        Dot product between user affinity vector and content genre vector.
        Works for ALL users including new ones (uses profile, not model).
        """
        if content_id not in self.content_genre_vectors:
            return 0.0
        
        user_vector = np.array([
            genre_affinities.get(genre, 0.1) for genre in GENRES
        ])
        content_vector = self.content_genre_vectors[content_id]
        
        # Normalize to 0-1 range
        raw_score = float(np.dot(user_vector, content_vector))
        max_possible = float(np.sum(content_vector))  # all genres match
        
        if max_possible == 0:
            return 0.0
        return min(1.0, raw_score / max_possible)
    
    def popularity_score(self, content_id):
        """Recency-based baseline score"""
        for item in self.content_catalog:
            if item["contentId"] == content_id:
                year = item.get("releaseYear", 2020)
                return min(1.0, (year - 2018) / 7.0)
        return 0.3
    
    # ─────────────────────────────────────────────
    # Main Ranking Method
    # ─────────────────────────────────────────────
    
    def rank(self, user_id, genre_affinities, recently_watched,
             variant="hybrid", limit=20):
        """
        Score and rank all content items for a given user.
        
        variant options:
          "hybrid"        → SVD + content-based (best for known users)
          "content-based" → content-based only (good for new users)
          "popularity"    → recency only (control/baseline)
        """
        recently_watched_set = set(recently_watched[-10:])
        scored = []
        
        is_known_user = user_id in self.known_users
        
        for item in self.content_catalog:
            content_id = item["contentId"]
            
            # Skip very recently watched content
            if content_id in recently_watched_set:
                continue
            
            # Calculate scores based on variant
            if variant == "popularity":
                final_score = self.popularity_score(content_id)
            
            elif variant == "content-based":
                final_score = self.content_based_score(
                    genre_affinities, content_id)
            
            else:  # hybrid (default)
                cb = self.content_based_score(genre_affinities, content_id)
                pop = self.popularity_score(content_id)
                
                if is_known_user:
                    # Known user: lean on SVD (it learned their specific taste)
                    svd = self.svd_score(user_id, content_id)
                    final_score = (0.5 * svd) + (0.3 * cb) + (0.2 * pop)
                else:
                    # Unknown user: SVD can't help, use content-based
                    final_score = (0.7 * cb) + (0.3 * pop)
            
            scored.append({
                "contentId": content_id,
                "score": round(final_score, 4)
            })
        
        # Sort descending by score
        scored.sort(key=lambda x: x["score"], reverse=True)
        
        return [item["contentId"] for item in scored[:limit]]

The Cold Start Problem — How MovieLens Helps
Here is a critical nuance. Your simulator generates users like user_001, user_002. MovieLens has users like ml_user_1, ml_user_2. These are completely different user IDs.
So when your system calls the ML service for user_001, the SVD model won't recognize them — they weren't in the training data.
This is actually fine, and here's why:
The SVD model handles MovieLens users perfectly. But for your synthetic users, the hybrid model gracefully degrades to content-based scoring (the is_known_user = False branch). This is intentional and correct behavior.
The real value of MovieLens in this architecture:
MovieLens data purpose:
  → Train the SVD model so it has learned latent factor representations
  → The item factors (V matrix) are now meaningful — each content item 
    has a rich 50-dimensional representation learned from real behavior
  → Even for unknown users, you can use item-item similarity from V

Your simulator purpose:
  → Generate user profiles (genre affinities) in Redis
  → Drive the content-based part of the hybrid model
  → Simulate realistic event patterns for demo
The two data sources serve different roles. MovieLens teaches the model what "good taste alignment" looks like across thousands of users. Your simulator creates the live profiles that the content-based layer uses.

What the Latent Factors Actually Capture
This is the part worth explaining in interviews because it's genuinely interesting.
After training on MovieLens, look at what SVD learned. You can examine the item factor matrix V:

    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

Toy Story and The Dark Knight have low similarity (family animation vs. dark crime). Toy Story and Sleepless in Seattle have higher similarity (both light, feel-good). The model learned this purely from watching which users rated both items similarly — it never saw the genre labels.
That's the power of collaborative filtering: it captures taste dimensions that no genre taxonomy can express.

How to Explain This in an Interview
When they ask "walk me through how your recommendation model works":

"The ML service implements a hybrid approach. For users who appear in the training data, I use matrix factorization — specifically Funk SVD trained on MovieLens ratings. The SVD decomposes the user-item interaction matrix into 50-dimensional latent factor vectors for both users and items. A user's predicted enjoyment of any item is the dot product of their user vector and the item's vector. For users not in the training data — which is all my synthetic users — the system falls back to content-based filtering using dot product similarity between the user's genre affinity profile and item genre vectors. The hybrid weights are 50% SVD, 30% content-based, 20% popularity for known users, and 70/30 content/popularity for unknown users."

When they ask "why use MovieLens if your users are synthetic?":

"MovieLens trains the item factor matrix. Even if a user is unknown to the model, the items now have rich latent representations learned from real human behavior — not just genre tags. The item similarity structure that SVD learns from MovieLens is more nuanced than anything you'd get from genre matching alone. In a production system, you'd retrain weekly on your own accumulated interaction data and phase out MovieLens entirely."

When they ask "how would you improve this?":

"Three directions. First, retrain on actual user interaction data as it accumulates — MovieLens is just a bootstrap. Second, add implicit feedback handling using the implicit library, which is designed for watch/skip data rather than explicit ratings. Third, explore neural collaborative filtering, which can capture non-linear relationships that SVD misses — but that's only worth the complexity at significant scale."
